# Phase 2: Forecasting & Regime Detection - Research

## Objective
Research the technical approach and implementation details required to plan Phase 2 of MLPO v1.0. This phase builds the core predictive models: the LSTM Volatility Forecaster and the Real-Time Regime Detector.

## Architectural Requirements from PRD

### 1. LSTM Volatility Forecaster (`models/lstm_covariance.py`)
- **Purpose:** Forecast the conditional covariance matrix dynamically without relying on static rolling windows.
- **Architecture:** 3-layer LSTM, 256 hidden units, Layer Normalization, Dropout (0.3).
- **Mathematical Constraint:** The predicted covariance matrix ($\hat{\Sigma}$) MUST be Positive Semi-Definite (PSD).
- **Implementation Strategy:** Instead of predicting $\hat{\Sigma}$ directly, the network predicts the lower triangular Cholesky factor $L$.
  - $\hat{\Sigma}(t) = L \cdot L^T + \epsilon I$
  - $\epsilon = 10^{-6}$ for numerical stability during backpropagation.
- **Hardware constraints:** Sequence length is strictly capped at 60 days to prevent memory overflow on 4GB VRAM.

### 2. Real-Time Regime Detection (`models/regime_detector.py`)
- **Purpose:** Continuously shift the risk budget based on macroeconomic leading indicators (VIX, TED spread, Yield Curve, Momentum, Liquidity, Market Cap).
- **Architecture:** 2-layer LSTM encoder (128 units) feeding into a Gaussian Mixture Model (GMM).
- **Soft Gating Logic:** Uses probabilities from the GMM to compute a dynamic risk budget $\beta_t = \sum_k (\gamma_k \cdot b_k)$.
- **Hard Override Logic:** If VIX $\ge 35$, the network's output is overridden to force a $\ge 40\%$ defensive asset allocation. This override must be integrated at the tensor level to maintain gradient flow or gracefully handle the discontinuity.

### 3. Testing Requirements (`tests/test_models.py`)
- Shape assertions for all tensor transformations.
- Gradient flow checks (asserting `.grad` is not `None` on layer 0 after `loss.backward()`).
- Covariance validity checks (`torch.linalg.eigvalsh(Sigma).min() > 0`).
- CUDA to CPU fallback routing checks.
- Deterministic execution checks using a centralized `seed_everything()` utility.

## Pitfalls & Landmines
- **Non-PSD Covariance:** Without Cholesky decomposition, the optimization step (Phase 3) will fail. The `torch.tril` function should be used to enforce the lower triangular structure.
- **Gradient Exploding in LSTM:** Proper layer normalization and dropout (0.3) are necessary, but we also must ensure AMP (Automatic Mixed Precision) compatibility for Phase 4.
- **Lookahead Bias in Regime Detection:** Ensure that state variables fed into the regime detector adhere to the `.shift(1)` rule established in Phase 1.
- **VIX Override Discontinuity:** A hard conditional `if VIX >= 35:` breaks gradients if not carefully implemented. It should be implemented using PyTorch tensor operations (e.g., `torch.where`) so computational graphs are preserved, even if gradients through the override path are zero.

## Validation Architecture
- **Unit Testing:** Implement robust Pytest cases targeting:
  1. `L * L^T` output shapes.
  2. The strict non-negativity of eigenvalues of the generated covariance matrix.
  3. Tensor shape matching for batch sizes and sequence lengths (e.g., `[batch_size, seq_len, num_assets]`).
- **Integration Validation:** Ensure the models can consume the PyTorch DataLoaders generated by `mlpo.data.dataset.MLPODataset`.
