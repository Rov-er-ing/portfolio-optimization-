**Target Environment:** Local Consumer GPU (NVIDIA RTX 3050 class) $\rightarrow$ Production Kubernetes (Future)
**Architecture Standard:** Differentiable Deep Learning with Convex Optimization (`cvxpylayers`)
**Asset Universe:** 45 Instruments (30 Equities, 10 Fixed Income, 5 Commodities) + 6 Macro State Variables

## 1. System Architecture & Mathematical Foundations
The system is an end-to-end differentiable pipeline. Gradients must flow seamlessly from the final portfolio loss metric back through the convex solver and into the feature extraction layers.

### 1.1 Module A: LSTM Volatility Forecaster
* **Purpose:** Forecasts the conditional covariance matrix dynamically without relying on static rolling windows.
* **Architecture:** 3-layer LSTM, 256 hidden units, Layer Normalization, Dropout (0.3).
* **Hardware Adaptation:** To prevent exploding gradients and memory overflow on 4GB VRAM, sequence length is strictly capped at 60 days.
* **Mathematical Constraint:** Covariance matrices must be positive semi-definite. The model will not output $\hat{\Sigma}$ directly. Instead, it predicts the lower triangular Cholesky factor $L$, computing the final matrix as:
    $$\hat{\Sigma}(t) = L \cdot L^T + \epsilon I$$
    *(where $\epsilon = 10^{-6}$ for numerical stability during backpropagation).*

### 1.2 Module B: Real-Time Regime Detection
* **Purpose:** Continuously shifts the risk budget based on macroeconomic leading indicators (VIX, TED spread, Yield Curve, Momentum, Liquidity, Market Cap).
* **Architecture:** 2-layer LSTM encoder (128 units) feeding into a Gaussian Mixture Model (GMM).
* **Logic:** Uses soft gating to compute the dynamic risk budget $\beta_t$:
    $$\beta_t = \sum_{k} (\gamma_k \cdot b_k)$$
* **Tail-Risk Override:** Hard constraint integrated at the tensor level. If VIX $\ge 35$, the network's output is overridden to force a $\ge 40\%$ defensive asset allocation.

### 1.3 Module C: Differentiable Risk Budgeting
* **Purpose:** Fuses forecasting and optimization. Solves for portfolio weights while allowing gradient flow.
* **Implementation:** `cvxpylayers`.
* **Optimization Problem:**
    $$\min_{w} \left\| \frac{w \odot (\hat{\Sigma}w)}{w^T \hat{\Sigma} w} - \beta_t \right\|^2 + \lambda_1 \|w\|_1 + \lambda_2 \|w - w_{t-1}\|^2$$
    *Subject to: $\sum w_i = 1, \quad 0 \le w_i \le 0.15$*
* **Hardware Adaptation:** `cvxpylayers` is highly VRAM intensive. Batch size during training must be restricted to 16 or 32, utilizing Automatic Mixed Precision (`torch.cuda.amp`) to compute the forward pass in FP16 while maintaining FP32 gradients.

### 1.4 Module D: Sparse Attention Mechanism
* **Purpose:** Focuses the model on the most relevant asset interactions while keeping computational complexity linear.
* **Logic:** Standard attention is $\mathcal{O}(p^2)$. By limiting attention strictly to an asset's $k$ most correlated neighbors ($k \in [10, 30]$), complexity drops to $\mathcal{O}(pk)$. Neighborhoods are cached during feature engineering to eliminate runtime re-computation.

## 2. Data Engineering & Integrity Protocols
Memory limitations dictate that the data pipeline cannot load the entire unified tensor into system RAM at once.

* **Lazy Loading:** PyTorch `Dataset` will utilize memory-mapped arrays or chunked HDF5 to load the 60-day rolling windows on the fly.
* **Temporal Integrity (Zero Lookahead):** All normalization (Z-scores) must be computed using exclusively trailing statistics ($t-1$ and earlier). Code will enforce `.shift(1)` universally before rolling computations.
* **Walk-Forward Embargo:** A mandatory 21-trading-day gap between the training window and validation/test windows. No data from this period is used for gradients or hyperparameter evaluation.

---

## 3. Training Strategy & Loss Topography
The system abandons standard MSE in favor of a composite financial loss function.

* **Composite Loss Equation:**
    $$\mathcal{L}(\theta) = -\frac{\mathbb{E}[R_p]}{\sqrt{\text{Var}(R_p)}} + \gamma_1 \text{MDD}(R_p) + \gamma_2 \text{Turnover} + \gamma_3 \|\theta\|_2^2$$
* **Hyperparameter Search:** Optuna (TPE Algorithm).
* **Hardware Adaptation:** Because 100 trials on a single consumer GPU will bottleneck, the pipeline will heavily leverage `Optuna.pruners.MedianPruner`. Any trial falling below the median Sharpe Ratio by epoch 20 will be aggressively terminated.

## 4. Software Engineering Standards
Every module written for this project will strictly adhere to the following quantitative engineering standards:

1.  **Type Hinting & Docstrings:** Exhaustive Python type hints and NumPy-style docstrings.
2.  **Financial Intuition Comments:** Inline comments explaining the *why* (e.g., "Annualizing standard deviation using $\sqrt{252}$ to match regime budget horizons").
3.  **Strict Testing:** Pytest classes for every module containing:
    * Shape assertions for all tensor transformations.
    * Gradient flow checks (asserting `.grad` is not `None` on layer 0 after `loss.backward()`).
    * Covariance validity checks (`torch.linalg.eigvalsh(Sigma).min() > 0`).
4.  **Device Agnostic:** Seamless `cuda` to `cpu` fallback using dynamic device routing.
5.  **Deterministic Execution:** Centralized `seed_everything()` utility for PyTorch, NumPy, and Python Random to ensure verifiable backtests.

## 5. Interpretability & API (Phase 5)
Regulatory-grade interpretability is non-negotiable. 

* **SHAP Integration:** DeepSHAP for the LSTM representations and KernelSHAP for the convex optimization layer.
* **Audit Logging:** Every JSON response from the API must include the regime state, VIX override flags, expected turnover, and a top-5 feature attribution list explaining exactly why a weight shifted.
* **FastAPI:** Asynchronous endpoints (`/v1/portfolio/optimize`) validated by Pydantic v2 schemas.
